# 1. 상태 관리란?

## 1-1. 상태(State)란?

상태(state)는 현재 가지고 있는 형태나 모양을 정의하면서, 변화할 수 있는 동적인 값을 말한다. 리액트 컴포넌트들은 모두 자신의 형태나 모양을 정의하는 state를 가지고 있다. 예를 들어 다음과 같은 것들이 상태에 해당된다.

✅ **사용자가 입력한 값** → 검색창의 입력값, 로그인 폼의 입력값

✅ **비동기 요청 결과** → API 응답 데이터

✅ **UI 상태** → 다크 모드 활성화 여부, 모달 열림/닫힘 여부

상태가 변화하면 **React는 이를 감지하고 컴포넌트를 다시 렌더링한다**. 따라서 상태를 어떻게 관리하느냐에 따라 성능과 유지보수성이 크게 달라질 수 있다.

---

### +) 왜 `let`, `const`를 안 쓰고 `state`를 사용할까?

→ 변수의 값이 바뀐다고 컴포넌트가 리렌더링 되지 않기 때문이다. React는 일반적인 변수가 아닌 state 값이 변화했을 때만 리렌더링 된다.

## 1-2. 전역 상태 vs 지역 상태

React에서는 상태를 크게 **지역(Local) 상태**와 **전역(Global) 상태**로 나눌 수 있다.

### **✅ 지역 상태 (Local State)**

**지역 상태는 특정 컴포넌트 내부에서만 사용하는 상태**를 의미한다. 주로 `useState` 또는 `useReducer`로 관리하며, 해당 컴포넌트에서만 상태를 변경하고 사용할 수 있다.

따라서 이 지역 상태를 부모나 자식 컴포넌트로 전달하고 싶다면 `props`를 사용해야 한다.

### **✅ 전역 상태 (Global State)**

**반면에 전역 상태는 여러 컴포넌트에서 공유하는 상태**를 의미한다. 따라서 여러 컴포넌트에서 접근이 가능하며 전역적으로 관리되기 때문에 데이터 일관성이 유지되는 특징이 있다.

# 2. Context API란?

## 2-1. Context API 개념

React의 **Context API**는 **컴포넌트 전체에서 전역적으로 데이터를 공유**할 수 있도록 도와주는 기능이다.

위에서 말했듯, 지역 상태의 변수를 사용할 때 기본적으로 React의 상태는 **`props`를 통해 전달**된다. 하지만 `props`는 오직 부모-자식에서만 상태를 전달할 수 있다는 단점이 존재한다.

### ❌ **Props Drilling 문제**

React에서는 부모-자식 관계를 통해 데이터를 전달해야 하는데, 계층 구조가 깊어질 경우 **불필요한 중간 컴포넌트들이 props를 전달하는 역할만 하게 되는 문제**가 발생한다. 이를 **Props Drilling**이라고 한다.

![Untitled](2%EF%B8%8F%E2%83%A3%2018dfec439d3680ff81f8d38028729fed/37f3c203-4691-4968-b27d-5554ffc2d41b.png)

Props Drilling이 발생하면 타이핑 양이 많아지는 것을 물론, `props`의 이름이 바뀐다면, 모든 컴포넌트를 찾아가서 이름을 바꾸어주어야한다.

<aside>

나도 현재 진행 중인 졸업 프로젝트에서 지도와 바텀시트가 함께 사용되는 UI를 만들고 있다.

- 구조:
    - 지도 화면이 최상위 부모이며, 그 안에 바텀시트가 포함되어 있다.
    - 바텀시트 내부에는 다양한 컨텐츠(컴포넌트)가 동적으로 변경된다.

지도에서 바텀시트에 어떤 컨텐츠를 띄울지 결정해야 하므로, props를 여러 단계에 걸쳐 전달해야 했다. 특히 지도 → 바텀시트 → 바텀시트 내부 컴포넌트로 이어지는 구조에서 props를 계속 전달해주어야 해 **Props Drilling**이 발생했다.

이러한 문제를 해결하기 위해 바텀시트 관련 상태들을 **전역적으로 관리**하기로 결정했다!

</aside>

### **💡 Context API를 활용하면?**

Context는 데이터들을 보관하는 일종의 데이터 보관소 역할을 하는 객체이다. 따라서 부모-자식 간 `props`를 여러 단계 거치지 않고도, **필요한 컴포넌트에서 직접 데이터를 가져올 수 있다.**

## 2-2. 언제 Context API를 사용하면 좋을까?

Context API는 **상태를 여러 컴포넌트에서 공유해야 할 때** 유용하다. 하지만 모든 전역 상태를 Context API로 관리하는 것은 좋은 선택이 아닐 수도 있다.

### **✅ Context API를 사용하면 좋은 경우**

✔️ **테마 (다크모드, 라이트모드 등)**

✔️ **사용자 인증 정보 (로그인 상태, 유저 정보)**

✔️ **다국어 지원 (언어 설정, 번역 데이터)**

✔️ **모달, 알림 상태 (전역적으로 열고 닫아야 할 UI 상태)**

Context API는 이처럼 **자주 변경되지 않지만, 여러 컴포넌트에서 필요**한 데이터를 관리할 때 적합하다.

### **❌ Context API를 사용하기 적절하지 않은 경우**

❌ **자주 변경되는 상태 (예: 입력 값, 실시간 API 응답 등)**

❌ **복잡한 비즈니스 로직이 포함된 경우**

## 2-3. 왜 자주 변경되는 값에서는 Context API를 사용하면 안될까?

리액트 컴포넌트들이 리렌더링 되는 상황은 다음과 같다.

1. 자신이 관리하는 state의 값이 변경되었을 때
2. 자신이 제공받는 props의 값이 변경되었을 때
3. 부모 컴포넌트가 리렌더링될 때

### **❌ Context API의 문제점**

따라서 Context API가 관리하고 있는 `value`가 변경되면 **모든 하위 컴포넌트가 리렌더링된다**. `useContext`를 사용한 모든 컴포넌트는 **필요하지 않아도 리렌더링될 수 있다.**

### **✅ 해결 방법**

1. **Context를 여러 개로 분리해 변경이 되는 Context와 변경이 되지 않는 Context로 나눠 불필요한 리렌더링을 방지한다.**
2. **`useMemo`를 활용해 value가 불필요하게 변경되지 않도록 최적화한다.**

따라서 Context API는 **전역 상태가 자주 변경되지 않을 때** 사용하는 것이 좋다! **자주 변경되는 전역 상태는 Redux, Zustand 같은 상태 관리 라이브러리를 고려하는 것이 더 적합할 수 있다.**

<aside>

졸업 프로젝트에서 바텀시트 상태를 전역적으로 관리하기로 결정한 후, 처음에는 Context API를 사용하려고 했다. 하지만 바텀시트가 단순한 UI 요소가 아니라, 현재 어떤 컨텐츠를 띄우고 있는지, 열려 있는 상태인지 등을 유지해야 했고, 특히 다른 페이지로 이동했다가 뒤로 가기를 누르면 바텀시트가 열려 있던 상태 그대로 돌아와야 하는 요구사항이 있었다.

그런데 Context API를 사용하면 바텀시트의 상태가 변경될 때마다 Context가 관리하는 값이 바뀌게 되고, React의 Context는 value가 변경될 때 이를 사용하는 모든 하위 컴포넌트를 리렌더링한다. 즉, 바텀시트가 열리고 닫힐 때마다 지도 페이지 전체가 불필요하게 리렌더링될 가능성이 높았다.

이러한 문제를 해결하기 위해 상태관리 라이브러리를 사용해보기로 결심했다!

</aside>

# 3. 상태 관리 라이브러리(Redux, Zustand, Recoil, Jotai)

## 3-1. 각 라이브러리의 개요 및 특징

### ✅ Redux

Redux는 가장 널리 사용되는 상태 관리 라이브러리 중 하나로, Flux 아키텍처를 기반으로 전역 상태를 관리한다.

- **특징**
    - `store`에서 전역 상태를 관리하고, `dispatch`를 통해 상태를 변경
    - 상태 변경은 **순수 함수인 Reducer**를 통해 이루어짐
    - **Redux Toolkit**을 사용하면 보일러플레이트 코드가 줄어듦
    - Middleware(Redux Thunk, Redux Saga 등) 활용 가능
- **장점**
    - 복잡한 상태 관리에 유용함 (대규모 애플리케이션에 적합)
    - 개발자 도구(DevTools) 지원이 강력함 (타임 트래블 디버깅 등)
    - 확장성과 유지보수성이 뛰어남
- **단점**
    - 보일러플레이트 코드가 많아 코드량이 증가
    - 작은 프로젝트에서는 다소 과한 선택이 될 수 있음
    - 상태 변경 과정이 다소 복잡할 수 있음

### ✅ Zustand

Zustand는 **React의 상태 관리를 단순하게 만들어주는 경량 상태 관리 라이브러리**다.

- **특징**
    - Redux보다 간단한 API를 제공
    - Context API보다 성능이 뛰어나며, 불필요한 리렌더링이 적음
    - `useStore` 훅을 통해 상태를 구독 및 변경 가능
    - 미들웨어와 DevTools 지원
- **장점**
    - 보일러플레이트가 거의 없음 (Redux 대비 코드가 간결함)
    - React의 useState처럼 쉽게 사용 가능
    - 전역 상태 변경 시 필요한 컴포넌트만 리렌더링됨
    - Immer, DevTools 등과 호환 가능
- **단점**
    - 대규모 상태 관리에는 부족할 수 있음
    - 공식 문서 및 커뮤니티가 Redux만큼 활발하지 않음

### ✅ Recoil

Recoil은 Facebook에서 개발한 상태 관리 라이브러리로, **React의 useState와 유사한 방식으로 전역 상태를 관리**한다.

- **특징**
    - `atom`과 `selector` 개념을 사용하여 상태 관리
    - React의 Concurrent Mode와 잘 호환됨
    - **컴포넌트 기반으로 상태를 나눠서 관리 가능**
- **장점**
    - React 개발 경험과 매우 유사한 방식으로 전역 상태를 다룰 수 있음
    - 부분적으로 상태를 구독할 수 있어 리렌더링 최적화 가능
    - 비동기 상태 관리 기능 내장 (Selector와 Suspense 활용 가능)
- **단점**
    - 공식적으로 아직 실험적인 라이브러리 (React 18 이후 정식 지원 여부 불확실)
    - Redux보다 생태계가 작아 확장성이 부족함
    - 서버 상태 관리에 적합하지 않음

### ✅ Jotai

Jotai는 **Recoil과 유사한 개념을 가진 상태 관리 라이브러리**로, 더 간결하고 가벼운 API를 제공한다.

- **특징**
    - `atom`을 사용해 상태를 정의하고, `useAtom`으로 상태를 읽고 변경 가능
    - 기본적으로 React의 useState처럼 동작하지만 전역 상태로 확장 가능
    - 비동기 상태 관리 기능 내장
- **장점**
    - 코드가 매우 간결하고 직관적
    - 리렌더링이 최소화됨
    - 사용법이 간단하며, Recoil보다 가벼움
- **단점**
    - 생태계가 작고, 미들웨어 지원이 부족함
    - 대규모 상태 관리에는 다소 부족할 수 있음

## 3-2. 어떤 상태 관리 라이브러리를 선택해야 할까?

| 선택 기준 | Redux | Zustand | Recoil | Jotai |
| --- | --- | --- | --- | --- |
| 프로젝트 규모 | 대규모 | 소/중규모 | 중/대규모 | 소/중규모 |
| 사용법 | 복잡 | 간단 | 중간 | 매우 간단 |

<aside>

나는 여러 상태 관리 라이브러리들 중에서 Zustand를 선택했다! 사실 Redux가 가장 많이 사용되는 라이브러리라 한 번 써보고 싶었지만, 졸업 프로젝트의 기한이 촉박해서 학습하는 데 많은 시간을 투자할 수 없었다. 대신, 학교 선배에게 Zustand가 사용법이 Context API와 가장 유사하면서도 간단하다는 이야기를 듣고 Zustand를 선택하게 되었다.

결과적으로 Zustand를 사용하면서 비교적 빠르게 전역 상태 관리를 적용할 수 있었고, 불필요한 리렌더링을 최소화하면서도 필요한 상태만 관리할 수 있어 만족스러웠다.

➡ 따라서, **프로젝트의 복잡도와 요구사항을 고려하여 적절한 상태 관리 라이브러리를 선택하는 것이 중요하다!** 

</aside>

# 결론

상태 관리 방식은 **지역 상태(Local State) 관리, Context API, 그리고 상태 관리 라이브러리(Redux, Zustand, Recoil 등)**로 나눌 수 있다.

단순한 상태는 `useState` 같은 지역 변수로 관리하는 것이 가장 효율적이며, 여러 컴포넌트에서 공유해야 하는 상태는 Context API를 활용할 수 있다.

그러나 Context API는 상태가 자주 변경될 경우 불필요한 리렌더링이 발생할 수 있어, 복잡한 상태 관리가 필요하다면 Redux, Zustand 같은 라이브러리를 고려하는 것이 좋다.

따라서, 프로젝트의 규모와 요구사항에 따라 적절한 상태 관리 방식을 선택하는 것이 중요하다.

단순한 전역 상태 공유라면 Zustand, Recoil, Jotai 같은 경량 라이브러리가 적합하며, 복잡한 상태 관리가 필요한 경우에는 Redux가 좋은 선택이 될 수 있다.

결국, **개발 효율성과 유지보수성을 고려해 프로젝트에 가장 적합한 상태 관리 방식을 선택**하는 것이 핵심이다!
